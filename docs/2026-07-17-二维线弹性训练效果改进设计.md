# 二维线弹性训练效果改进设计

## 1. 背景与目标

二维线弹性悬臂梁闭环已经完成真实 FEniCSx 数据生成、物理门禁、POD-RBF、Vector DeepONet、开发评价、可视化和报告。现有 Smoke 运行状态为 `development_complete`，证明流程正确走通，但尾部误差尚不理想：

| 指标 | Vector DeepONet | POD-RBF | Full 数值门槛 |
|---|---:|---:|---:|
| 全场相对 L2 中位 | 1.75% | 3.18% | ≤ 3% |
| 全场相对 L2 P95 | 15.91% | 16.69% | ≤ 8% |
| 全场相对 L2 最差 | 61.89% | 63.41% | ≤ 15% |
| 自由端误差 P95 | 18.92% | 17.87% | ≤ 8% |
| 柔度误差 P95 | 4.59% | 43.28% | ≤ 10% |

本轮目标是在不新增 FEniCSx 样本、不读取任何 Full 封存测试的前提下，复用现有 `96/24/24` Smoke 数据，优先消除载荷方向表示和训练目标失配造成的尾部误差。

本轮把现有 Full 门槛作为开发阶段的数值目标，但即使全部达到，结果仍只能称为 Smoke 开发证据，不得标记为 `accepted` Full，也不得通过正常可信推理入口冒充确认性模型。

## 2. 现有证据与根因判断

使用现有 Smoke 数据和同一随机种子在内存中复现当前训练，得到与已保存报告一致的最佳 epoch、验证损失和开发测试指标，说明问题稳定可复现，不是偶发设备波动。

逐样本诊断显示：

- 最差样本为 `development_test-00133-83199b45fb86`，载荷角 `theta=0.1043`，即近水平载荷。
- 该样本 DeepONet 与 POD-RBF 的全场相对 L2 分别约为 `61.89%` 和 `63.41%`。
- DeepONet 逐样本误差与 `|sin(theta)|` 的 Spearman 相关系数为 `-0.489`；越接近水平载荷，越容易出现大误差。
- 误差与最近训练样本特征距离的 Spearman 相关系数只有 `0.146`，最差样本也不是覆盖距离最远的样本。
- 96 个训练样本中已有 14 个 `|sin(theta)|<0.2` 的近水平工况，其中 7 个小于 `0.1`，因此当前首要问题不是完全缺少水平载荷样本。
- 当前 POD 按全局 `99.9%` 能量选模时只保留 1 个模态；开发测试的 POD 投影上限本身为中位 `1.53%`、P95 `12.90%`、最差 `40.36%`，说明高能量弯曲响应掩盖了低能量水平拉伸响应。

线弹性对载荷向量严格线性。当前网络虽然把 `cos(theta)` 和 `sin(theta)` 作为普通 Branch 特征，却允许 MLP 对它们作任意非线性混合。竖向载荷产生的弯曲位移远强于水平载荷产生的拉伸位移；近水平工况中只要泄漏少量竖向响应，相对误差就会被显著放大。

因此当前根因优先级为：

1. 载荷方向线性关系未被架构严格编码，产生方向串扰。
2. 全局能量和平均验证损失偏向高能量弯曲工况，尾部风险权重不足。
3. POD 诊断基线使用单一全局能量阈值，不能公平诊断水平与竖向响应。
4. 参数覆盖可能影响剩余误差，但现有证据不支持把它视为第一根因。

## 3. 方案比较

### 3.1 立即增加随机 FEniCSx 样本

优点是增加参数覆盖；缺点是成本较高，而且不能从结构上消除方向串扰。当前最差样本并非距离训练集最远，直接增加随机 Latin Hypercube 样本可能只是用更多数据缓解错误表示。

本轮不采用该方案作为第一步。

### 3.2 只增加 epoch、网络宽度或修改损失

优点是改动较小；缺点是网络仍需从数据近似学习本来已知的严格线性关系。更多训练可能继续改善平均误差，却不能保证近水平工况没有竖向响应泄漏。

本轮保留损失和预算改进作为第二阶段，不把它作为首个变量。

### 3.3 精确载荷方向分解并逐项消融

把网络输出改为水平单位载荷响应基和竖向单位载荷响应基，再由物理参数中的 `cos(theta)`、`sin(theta)` 精确组合。该方案直接对应根因，同时保留现有 `P/E` 和固支边界硬约束。

这是本轮推荐并批准的主路线。

## 4. 模型架构设计

### 4.1 精确方向叠加

将网络学习目标改写为：

\[
\widehat{\mathbf u}(x,y)
=\frac{P}{E}\frac{x}{L}
\left[
\cos\theta\,\mathbf h(\nu,y_0,w;x,y)
+\sin\theta\,\mathbf v(\nu,y_0,w;x,y)
\right],
\]

其中：

- `h=(h_x,h_y)` 表示水平单位载荷的二维响应基；
- `v=(v_x,v_y)` 表示竖向单位载荷的二维响应基；
- Branch 只接收 `(nu,y0,w)` 三维特征；
- Trunk 继续接收归一化 `(x,y)`；
- 通用 `VectorDeepONet` 输出四个通道 `(h_x,h_y,v_x,v_y)`；
- 悬臂梁约束层负责方向组合、`P/E` 缩放和 `x/L` 固支约束。

由此得到以下严格性质：

- `theta=0` 时竖向响应基的贡献精确为零；
- `theta=±pi/2` 时水平响应基的贡献精确为零；
- `theta+pi` 的预测精确等于原预测的相反数；
- 任意角度预测精确满足水平、竖向单位载荷的线性叠加；
- `P` 加倍、`E` 加倍和 `x=0` 的现有硬约束继续成立。

模型规格增加显式架构版本 `directional_linear_v2`。模型文件、网络结构、运行身份和清单必须记录该版本，防止旧的双输出权重被新代码误加载。

### 4.2 训练数据和归一化

训练仍使用原始混合方向位移场，不尝试从单个样本伪造独立的水平/竖向标签。方向基通过所有不同角度样本的联合监督被识别。

模型 Branch 归一化只在训练集上拟合 `(nu,y0,w)`。POD-RBF 可继续使用包含方向的特征，但其诊断表示将在第三阶段改为方向平衡形式。测试集不参与任何归一化、结构选择或早停。

### 4.3 损失与选模的分阶段原则

第一阶段只替换方向表示，保持当前样本、随机种子、网络宽度、潜维、学习率、epoch、坐标子批和 `balanced_relative_dimensionless_shape_mse_v1` 不变，以隔离架构变化的因果效果。

若第一阶段没有达到开发数值目标，第二阶段才引入以下两个独立变化，并分别形成消融结果：

1. 在训练批次内计算逐样本相对形状损失，使用“全部样本均值 + 最差 25% 样本均值”的固定组合，尾部项权重为 `1.0`。
2. 早停和候选种子选择改用完整验证网格的固定风险分数：`median_relative_l2 + p95_relative_l2 + worst_relative_l2`；相同分数时按种子升序选择。

两个变化不得与方向架构在同一次首次实验中同时引入。每次实验只改变一个假设，失败时回到上一已验证检查点。

## 5. 物理一致的数据增强

只有方向架构实验仍存在明显尾部误差时，才加入关于梁中线的镜像增强：

```text
y0' = 1 - y0
theta' = -theta
u_x'(x,y) =  u_x(x,1-y)
u_y'(x,y) = -u_y(x,1-y)
```

观测网格必须关于 `y=0.5` 对称，字段重排必须通过坐标身份验证。镜像样本只从训练集生成；验证和开发测试保持原样。增强记录原样本 ID 和变换类型，不能伪装成新的 FEniCSx 求解样本。

载荷反向 `theta+pi → -u` 已由 `directional_linear_v2` 精确保证，不再复制一份完全冗余的训练样本。

## 6. 现有数据的审计式复用

当前运行身份同时绑定请求和完整规格；修改模型后不能覆盖原 `development_evaluation.json`，也不应重新调用 FEniCSx 计算完全相同的样本。

新增仅限 Smoke 的数据复用入口：

```powershell
uv run surrogate-loop elasticity2d run \
  --config examples/elasticity_2d_cantilever/smoke.json \
  --reuse-data-from runs/elasticity-smoke-b694306dfbc8 \
  --runs-dir runs \
  --request "复用已验证数据训练 directional_linear_v2"
```

复用规则：

- 只允许源运行和目标配置均为 Smoke；Full 禁止使用该入口。
- 源运行必须具有完整的请求身份、求解器清单、开发数据、开发测试数据和 SHA-256。
- 问题、网格、观测坐标、采样种子、样本 ID、角色、六列参数、求解器门槛和数据协议必须与目标配置一致。
- 模型、训练和评价配置允许变化，并进入新的运行身份。
- 数据文件复制到新的运行目录并重新校验摘要；原运行目录保持只读，不覆盖历史报告。
- 新运行记录源运行绝对路径、源数据摘要、目标数据摘要和复用原因。
- 任何身份或摘要不一致都立即失败，不能回退为静默重新求解。

该入口的目的是让模型开发实验可复现、可审计，同时避免不必要的有限元计算。它不允许跨问题、跨网格或跨采样复用。

## 7. POD-RBF 诊断改进

POD-RBF 仍只作为诊断基线，不能替代 DeepONet 验收。其改进在方向架构之后单独实施：

1. 先除以 `P/E` 得到无量纲形状场。
2. 分别按水平权重 `|cos(theta)|` 和竖向权重 `|sin(theta)|` 构造平衡后的训练矩阵。
3. 对两个方向响应使用独立的 POD 能量统计和模态上限，每个方向至少保留 1 个模态。
4. 报告水平、竖向及混合方向的投影上限和插值误差，避免单个全局能量数字掩盖低能量响应。

POD 改进用于解释数据可学习性，不与首次 DeepONet 架构消融捆绑。

## 8. 实验顺序和停止条件

### 阶段 A：冻结当前基线

- 自动化重现当前模型输出形状、硬约束和损失合同。
- 记录现有开发测试指标和最差样本身份。
- 不修改现有运行目录。

### 阶段 B：方向线性架构

- 以测试驱动方式实现 `directional_linear_v2`。
- 复用相同数据、相同训练预算和相同随机种子训练。
- 比较全部六项指标、逐角度误差和最差样本。

### 阶段 C：尾部风险训练

仅当阶段 B 未达到开发数值目标时执行。先加入固定尾部损失，再单独加入风险分数早停与选模；每步都重跑同一数据并保存消融结果。

### 阶段 D：镜像增强与 POD 诊断

仅当阶段 C 后仍有明显尾部误差时执行。镜像增强只作用于训练集；POD 方向平衡单独报告。

### 阶段 E：是否新增 FEniCSx 样本的决策

满足以下任一条件时，停止无数据增量实验并报告需要定向补样本：

- `directional_linear_v2`、尾部风险训练和镜像增强均完成后，P95 仍高于 `8%` 或最差仍高于 `15%`；
- 误差开始与训练集覆盖距离呈明显正相关，Spearman 系数达到 `0.5` 以上；
- 最差样本稳定集中在 `(nu,y0,w)` 参数域边界，且架构消融不再改善。

届时新增样本应按失败区域定向设计，不能直接扩大随机 LHS 后把旧开发测试结果当成独立证据。

## 9. 验收与报告

本轮主要开发目标沿用 Full 的数值门槛：

- 全场相对 L2 中位不超过 `3%`；
- P95 不超过 `8%`；
- 最差不超过 `15%`；
- 自由端误差 P95 不超过 `8%`；
- 柔度误差 P95 不超过 `10%`；
- 固支边界最大绝对误差不超过 `1e-7`；
- 单样本 CPU 加速仍至少 `100` 倍。

此外必须报告：

- 与原 Smoke 完全相同开发测试样本上的逐样本对照；
- 近水平、斜向、近竖向三个方向区间的误差统计；
- 每次消融只改变的单一因素；
- 源数据运行目录和全部复用摘要；
- 训练种子、最佳 epoch、停止原因、训练时间和显存；
- 当前证据仍为 Smoke，未读取或消费 Full 封存测试。

如果数值目标未全部达到，但 P95 和最差误差相对原 Smoke 至少各下降 `50%`，且中位误差不高于 `3%`，可以把结果记录为“结构性改进有效但仍需补样本或继续建模”，不能标记为已完成确认性精度目标。

## 10. 测试与安全边界

实现必须遵循测试驱动开发，至少覆盖：

- 四通道方向基的输出形状和梯度；
- `theta=0`、`±pi/2`、任意角度和 `theta+pi` 的精确叠加；
- `P/E`、固支边界和非有限输入保护；
- 三维 Branch 归一化只使用训练集；
- 旧双输出权重不能按 `directional_linear_v2` 加载；
- Smoke 数据复用的身份、摘要、复制和来源记录；
- Full、跨网格、跨采样、损坏摘要和失败源运行全部拒绝复用；
- 镜像增强的参数、坐标、分量符号和样本来源；
- 尾部损失和风险分数的确定性；
- 标量、一维热传导、假求解 E2E、真实微型 FEniCSx E2E 和现有保护机制无回归。

本轮禁止：

- 启动二维线弹性 Full；
- 放宽 `3%/8%/15%` 等既有门槛；
- 修改或覆盖原 Smoke 运行产物；
- 把增强样本表述为新的 FEniCSx 样本；
- 因最差样本失败而删除该样本；
- 在没有摘要和身份匹配的情况下复用数据。

## 11. 完成定义

本轮完成需要同时满足：

1. 根因和基线可由自动化诊断稳定复现。
2. `directional_linear_v2` 精确满足方向线性、`P/E` 和固支约束。
3. 相同 Smoke 数据通过新运行身份完成审计式复用，原运行不被覆盖。
4. 每个实验只改变一个因素，并保存可比较指标。
5. 达到开发数值目标，或根据预先定义的停止条件形成定向补样本建议。
6. Ruff、普通 pytest、FEniCSx 求解器测试和真实微型跨环境测试全部通过。
7. 文档准确说明最终状态、证据等级、数据复用和未执行 Full 的边界。
