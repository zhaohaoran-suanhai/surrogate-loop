# Agent 协作指南

本指南规定新对话如何理解、演示、运行和扩展 surrogate-loop。目标是让不熟悉神经网络训练的用户始终知道当前问题、正在做什么、证据到哪一级，以及下一步需要什么授权。

## 1. 只读接管

新 Agent 先执行不改变仓库和外部状态的接管：

1. 读取仓库根目录的 `AGENTS.md`。
2. 读取[项目文档地图](../README.md)。
3. 读取[当前能力与状态](../当前能力与状态.md)。
4. 查看 `git status --short` 和最近提交。
5. 把请求归类为演示、运行已有闭环、诊断或开发新 PDE。

只读接管不授权安装依赖、生成训练数据、修改配置、启动 Full、提交代码或删除运行产物。不要为了“了解背景”默认全文读取所有历史实施计划。

## 2. 任务分类

| 用户意图 | 任务路由 | 下一份文档 |
|---|---|---|
| “介绍价值”“做个演示” | 演示 | 状态页；随后进入 `docs/demos/` 的对应手册 |
| “运行这个算例”“从头训练” | 运行已有闭环 | 对应操作指南和配置 |
| “为什么失败”“环境是否正常” | 诊断 | 环境指南、对应操作指南、运行状态与错误文件 |
| “增加新的方程/几何/输出” | 开发新 PDE | 中文设计规格和新的中文实施计划 |

请求同时包含多类任务时，先只读接管和解释，再按“确认范围 → 运行或开发”的顺序推进。

## 3. 演示路径

演示默认采用快速展示：解释自然语言如何收敛为白名单配置，展示已有数值数据、训练报告、误差图和推理速度，再准确说明当前证据是 Smoke 还是 Full。

二维线弹性悬臂梁是推荐主线，因为它同时展示：

- 明确的控制方程、边界条件和材料参数；
- 独立 FEniCSx 求解环境生成高保真数据；
- Vector DeepONet 学习参数到二维位移场的算子；
- 物理门禁、模型指标、可视化和可信状态边界。

只有用户希望现场证明计算链路、并接受状态页所列本机成本时，才切换为从头运行 Smoke。演示不能把本地不存在的 `runs/` 图像说成已经展示，也不能把二维线弹性 Smoke 说成 Full。

## 4. 运行已有闭环

先选择白名单配置并运行 `validate`，再执行训练命令。三个入口分别是：

- [标量 ODE](标量闭环操作指南.md)
- [一维热传导](一维热传导闭环操作指南.md)
- [二维线弹性](二维线弹性闭环操作指南.md)

Agent 在运行前必须说明使用哪个配置、会生成什么、预计成本来自哪里、结果属于哪个证据等级。运行中持续播报阶段。完成后读取命令返回的真实运行目录，不能猜测运行标识。

二维线弹性从头运行默认指 Smoke。Full 不是 Smoke 的自动下一步，必须在 calibration 和 Smoke 评审后重新获得用户明确确认。

## 5. 诊断路径

先判断失败属于哪一层：

1. 环境：uv、PyTorch、CUDA、Conda、FEniCSx 或编译工具不可用。
2. 配置：schema、参数域、预算或运行角色不合法。
3. 协议：JSON/NPZ 版本、SHA-256、身份或跨环境交换失败。
4. 求解器：收敛、残差、力平衡、网格或制造解门禁失败。
5. 训练：OOM、NaN/Inf、预算耗尽或优化未收敛。
6. 验收与推理：指标未达标、状态非 accepted、摘要损坏或输入域外。

诊断时先读取 `error.json`、状态、阶段摘要和对应诊断文件，再给出根因证据。不得静默删除失败样本、修改请求来逃避失败、降低 Full 门槛或绕过哈希验证。

## 6. 开发新 PDE

新 PDE 不是把任意方程文本直接交给运行时执行。开发顺序固定为：

```text
自然语言需求
→ 明确物理域、参数、输入输出和边界条件
→ 中文设计规格评审
→ 中文实施计划评审
→ 失败测试与最小实现
→ 求解器校准
→ Smoke 开发闭环
→ 经再次授权的 Full 确认性验收
```

规格至少要回答：方程和本构、几何、边界与载荷、参数域、求解器、数据协议、代理模型、数据划分、物理门禁、验收指标、算力预算、失败恢复和可信推理边界。未经规格通过，不直接开始编写新的 PDE 核心代码。

## 7. 操作前说明合同

每次有成本或会修改状态的操作前，Agent 用简短自然语言说明：

```text
为什么执行：本步骤要验证或产生什么。
输入与输出：使用哪个配置/目录，会新增哪些产物。
预计成本：时间、CPU/GPU、内存和磁盘估计，以及估计来源。
证据等级：本步骤属于假求解 E2E、真实 E2E、calibration、Smoke 或 Full。
授权边界：是否触及 Full、安装依赖、外部写入或其他需确认动作。
```

无法从状态页、指南或校准报告得到可靠估计时，应明确说“尚无本机估计”，不能编造精确时间。

## 8. 进度播报合同

长于一分钟或含多个阶段的操作，使用统一播报：

```text
已完成：刚刚完成的阶段和可核验结果。
正在执行：当前阶段及其目的。
当前证据：到目前为止证明了什么、还没有证明什么。
下一步：紧接着的动作、预计成本或需要的授权。
```

二维线弹性从头演示使用六阶段编号。若发生恢复或复用，必须在“已完成”中写明复用的阶段和身份校验，不能称为完全从零运行。

## 9. 完成报告合同

完成报告必须包含：

- 实际最终状态，例如 `accepted`、`rejected`、`development_complete` 或 `failed`；
- 关键指标的通俗解释，而不只罗列数字；
- 可核验证据路径，包括配置、运行目录、报告和图像；
- 证据等级与限制，特别是 Smoke/Full、域内/域外和网络输出/求解器诊断的区别；
- 可选下一步及其成本和授权要求。

面向非专业受众时，先给结论，再解释公式、误差和加速比各自回答什么问题。

## 10. 禁止行为与授权边界

- 不执行自然语言生成的任意方程或 Python 代码；只使用已审查配置和固定 CLI。
- 不把假求解 E2E 当作有限元科学证据。
- 不把开发测试上的 Smoke 表现当作 Full 确认性验收。
- 不读取封存测试来调参，不事后放宽验收门槛。
- 不绕过 SHA-256、状态或输入域校验来加载模型。
- 不把 `runs/`、训练数据或模型提交 Git。
- 不在用户未明确确认时启动二维线弹性 Full。
- 不在用户只要求解释、审查或诊断时擅自修改代码或运行高成本训练。
