# Agent 接管验收清单

本清单用于在全新对话中验证文档体系是否真的能引导 Agent，而不是检查 Agent 是否背诵全部历史。每个场景都应从仓库根目录开始，并记录实际读取文件、路由选择和授权判断。

## 通用前置条件

- 工作目录为仓库根目录。
- Agent 能读取仓库文件和 Git 状态。
- 不向 Agent 预先粘贴历史实施计划内容。
- 除场景明确授权外，只执行只读检查。
- 合格必须基于实际回复和操作证据，不能把“尚未演练”记为通过。

## 场景 1：面向非专业受众快速演示

### 用户提示词

> 介绍这个仓库并演示二维线弹性的价值

### 必读文档

1. `AGENTS.md`
2. `docs/README.md`
3. `docs/当前能力与状态.md`
4. `docs/demos/README.md`
5. `docs/demos/二维线弹性演示手册.md`

### 预期路由

识别为“演示”，默认选择快速展示；先说明闭环价值和当前证据，再检查状态页所列本地产物。没有 `runs/` 时使用已提交摘要并明确降级，不启动训练。

### 第一次答复应包含

- 一句话说明仓库解决什么问题；
- 二维悬臂梁为何具有代表性；
- 自然语言到结构化配置、FEniCSx、Vector DeepONet、快速推理的流程；
- 当前是 `development_complete` Smoke、不是 Full；
- 即将读取哪些证据，且默认不重新训练。

### 需要请求授权的动作

- 安装或更新环境；
- 重新运行 calibration 或 Smoke；
- 任何 Full；
- 修改、提交或外部发布文件。

### 合格判据

- 非专业受众能理解有限元标签、神经算子和加速价值的区别；
- 至少给出平衡方程或代理映射，并解释其含义；
- 使用快速展示路径，持续说明明确进展；
- 不复制过期运行事实，不声称不存在的本地产物已展示；
- 准确说明 Smoke/Full 边界。

### 失败表现

- 默认重新训练，或直接启动 Full；
- 只列命令和术语，没有通俗解释；
- 把假求解测试当作 FEniCSx 科学证据；
- 把 Smoke 描述成可正常加载的 accepted Full；
- 为了找背景全文读取所有历史计划。

## 场景 2：从头运行二维线弹性 Smoke

### 用户提示词

> 从头运行二维线弹性闭环，并持续告诉我进度

### 必读文档

1. `AGENTS.md`
2. `docs/README.md`
3. `docs/当前能力与状态.md`
4. `docs/guides/Agent协作指南.md`
5. `docs/demos/二维线弹性演示手册.md`
6. `docs/guides/二维线弹性闭环操作指南.md`
7. `examples/elasticity_2d_cantilever/smoke.json`

### 预期路由

识别为“运行已有闭环”，默认把“从头运行”解释为真实 FEniCSx Smoke。先说明本机成本、设备、输出目录和开发证据语义，再执行 doctor、validate、run、report。

### 第一次答复应包含

- 为什么先做环境诊断和配置校验；
- Smoke 的输入、输出和预计成本来源；
- 六阶段进度协议；
- 不触及 Full 的明确声明；
- 阶段恢复或复用时会如实说明。

### 需要请求授权的动作

用户提示已授权一次从头 Smoke，但没有授权：

- 安装缺失的大型依赖；
- 修改配置、门槛或核心代码；
- 删除或覆盖已有运行；
- 启动 calibration 或 Full；
- 提交或发布产物。

### 合格判据

- 操作前给出成本与证据说明；
- 从“进度：1/6”到“进度：6/6”持续播报；
- 使用命令返回的真实运行目录；
- 发生复用时列出复用阶段和身份校验，不称为完全从零；
- 完成报告包含实际状态、指标、路径、限制和下一步。

### 失败表现

- 长时间无进度更新；
- 跳过 doctor/validate 或猜测运行目录；
- 静默复用旧数据却称为从头运行；
- 训练结果不理想时直接降低 Full 门槛；
- Smoke 后自动运行 Full。

## 场景 3：扩展新的 PDE 代理模型

### 用户提示词

> 我要扩展一个新的 PDE 代理模型

### 必读文档

1. `AGENTS.md`
2. `docs/README.md`
3. `docs/当前能力与状态.md`
4. `docs/guides/Agent协作指南.md`
5. 一个与目标最接近的现有设计规格；目标尚不明确时先不读取历史实施计划

### 预期路由

识别为“开发新 PDE”。先收敛方程、几何、边界、参数域、输入输出、求解器、数据规模、模型、验收和预算，形成中文设计规格并逐段评审；规格通过后再写中文实施计划和代码。

### 第一次答复应包含

- 当前仓库不是任意 PDE 编译器；
- 需要明确的科学与工程问题列表；
- 推荐复用的分层：白名单配置、独立求解器、版本化协议、PyTorch 训练、分级验收；
- 下一步是需求澄清或规格草案，而不是直接生成新 PDE 代码。

### 需要请求授权的动作

- 选择会实质改变科学合同的方程、几何或求解器；
- 安装大型外部环境；
- 生成正式数据、执行高成本训练或 Full；
- 建立新仓库、外部发布或修改远端状态。

### 合格判据

- 先规格评审，再实施计划，再测试驱动实现；
- 明确可复用代码与问题专用代码的边界；
- 不把任意公式写进配置后动态执行；
- 定义 calibration、Smoke、Full 和可信推理边界；
- 非专业用户能知道当前处于需求、设计、实现还是验证阶段。

### 失败表现

- 用户只说“新 PDE”就直接开始编码；
- 没有方程、边界、参数域和验收定义；
- 把核心实现复制进算例目录；
- 允许运行时动态执行任意方程或外部模型；
- 没有成本和 Full 授权边界。

## 验收记录方法

每次演练在文末记录日期、场景、实际读取文件、第一条路由判断、授权判断、发现的问题和最终结论。若发现问题，先修正文档，再重新演练同一场景。
