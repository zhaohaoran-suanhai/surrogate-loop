# 标量代理模型最小闭环设计

## 1. 建设目标

建立一个最小、可审计的代理模型训练工作流，用它验证以下命题：自然语言请求可以安全地驱动一套确定、可验证、可复现的代理模型训练工具链。

第一版仅支持以下固定问题：

\[
\frac{du}{dt}=\gamma u+0.5t,\qquad
\gamma\in[-1,1],\qquad
t\in[0,1],\qquad
u(0)=0,
\]

预测合同固定为：

\[
\gamma\longmapsto u(1).
\]

第一版由 Codex 充当自然语言入口。Codex 负责把用户已经确认的需求转换为结构化配置，并调用固定的 Python 命令行工具。运行时不解析任意方程，不执行模型生成的训练代码，也不把自然语言直接交给数值求解器。

## 2. 方案比较

### 方案 A：工作流优先的标量代理模型——已选定

只预测 `gamma -> u(1)`，比较多项式响应面、Gaussian Process Regression（GPR）和小型多层感知机。该方案规模最小，但仍覆盖配置校验、数值数据生成、防数据泄漏划分、候选模型比较、产物保存、模型重载、域外拒绝和自然语言调度等完整环节。

### 方案 B：完整时间轨迹代理模型

预测 `gamma -> [u(t_1), ..., u(t_101)]`，推荐先使用 PCA/POD 降维，再预测低维系数。该方案更接近场代理模型和神经算子，但会在工作流尚未验证前引入降维、重构误差和轨迹可视化等额外问题。

### 方案 C：神经算子优先

直接使用 DeepONet 或坐标型神经网络。该方案与 PDEFlow 更接近，但会同时引入 PyTorch、网络结构、GPU 可重复性、训练时间和更复杂的失败诊断，不能提高第一阶段最需要的闭环证据。

第一版采用方案 A。方案 B 是标量闭环稳定后的下一个科学里程碑；方案 C 不进入本轮实施范围。

## 4. 第一版范围

### 4.1 范围内

- 一个白名单 ODE 模板，方程结构不可由用户任意修改。
- 使用 Pydantic 定义参数域、采样数量、数据划分、候选模型、随机种子和验收标准。
- 使用 `scipy.integrate.solve_ivp` 生成数值标签。
- 使用独立解析解校验数值标签。
- 按不同的 `gamma` 工况划分训练集、验证集和测试集。
- 三类候选模型：多项式响应面、GPR 和 `MLPRegressor`。
- 使用验证集选模，只对最终模型进行一次测试集评估。
- 保存预处理器、模型、指标、清单、模型卡和诊断图。
- 在新 Python 进程中重载模型并推理。
- 拒绝非有限输入和训练参数域之外的输入。
- 提供配置校验、冒烟运行、正式运行、报告查看和预测命令。
- 使用 pytest 验证科学正确性、流程不变量、模型持久化和端到端冒烟流程。

### 4.2 范围外

- 任意方程解析。
- `eval`、`exec`、动态执行生成代码或加载外部不可信模型文件。
- FEniCSx、CFD/FEM、DeepONet、FNO、PINN 和 PyTorch。
- Web 页面、外部 LLM API、运行时多 Agent 系统和 Agents SDK。
- 多轮 JSON Patch 和规范状态管理。
- Optuna、MLflow、分布式训练、GPU 训练和生产部署。
- 对训练参数域之外的数据自动外推。

## 5. 总体架构

项目采用 `src` 布局，将科学计算、机器学习和流程编排分离：

```text
用户自然语言请求
       ↓
Codex 生成受约束的 spec.json
       ↓
Pydantic 确定性校验
       ↓
固定 Python CLI
       ↓
数值数据生成 → 数据划分 → 候选模型训练 → 验证集选模
       ↓
测试集最终评估 → 验收门 → 独立运行目录
       ↓
模型重载 → 参数域检查 → u(1) 预测
```

CLI 只负责流程编排。配置校验、解析解、数值求解、数据划分、模型构建、评价、产物写入和推理边界检查分别放在职责单一的模块中。

## 8. 组件职责

### 8.1 配置合同

`config.py` 使用 Pydantic 并拒绝未知字段。问题标识固定为 `forced_reaction_scalar_endpoint_v1`，不接受可执行方程字符串。配置必须校验有限且有序的参数范围、正整数样本数量、受支持模型名称和明确验收阈值。

正式配置使用随机种子 `20260716`，包括 120 个训练工况、40 个验证工况和 40 个测试工况。冒烟配置使用相同种子，包括 24 个训练工况、8 个验证工况和 8 个测试工况。两者均通过 NumPy 的种子随机数生成器在 `[-1,1]` 上均匀采样，求解前拒绝重复浮点值。

### 8.2 科学问题和数据生成

`domain.py` 提供 ODE 右端项和解析解。当 `gamma = 0` 时使用连续极限 `u(t)=t^2/4`；当 `abs(gamma*t) < 1e-4` 时使用级数展开，其余情况使用 `expm1` 形式，避免接近零时发生消减误差。

`data.py` 使用 `solve_ivp(method="DOP853", rtol=1e-10, atol=1e-12)` 求解每个工况并提取 `u(1)`。每个数值标签与解析结果的绝对误差和相对误差都必须不超过 `1e-9`。求解失败、非有限结果或误差超限都会在训练前终止本次运行。

### 8.3 数据划分

`split.py` 在任何预处理前按不同 `gamma` 工况建立训练、验证和测试分区。同一个 `gamma` 不能出现在多个分区。归一化器和其他预处理只能在训练集上拟合。

### 8.4 模型注册表和选模

`models.py` 构造以下 scikit-learn Pipeline：

- 1、2、3 阶多项式特征，随后使用 `Ridge(alpha=1e-8)`；
- `GaussianProcessRegressor`，使用 `ConstantKernel * RBF`、`alpha=1e-10`、`normalize_y=True`、2 次优化器重启和配置随机种子；
- `MLPRegressor`，使用隐藏层 `(32,32)`、`tanh`、`alpha=1e-4`、`max_iter=2000`，并从训练分区内部拿出 20% 用于 early stopping。

候选模型使用验证集 NRMSE 排名。NRMSE 定义为 RMSE 除以 `max(y_validation)-min(y_validation)`；验证目标范围为零时判为配置或数据错误。模型还要记录 MAE 和最大绝对误差。NRMSE 差异不超过 `1e-12` 时，按 1 阶 PRS、2 阶 PRS、3 阶 PRS、GPR、MLP 的顺序优先选择更简单模型。测试集只用于最终选中模型的一次评估。

### 8.5 验收和产物

`evaluation.py` 要求测试集 NRMSE 不高于 `0.03`，最大绝对误差不高于 `0.01`。完整运行可以得到 `accepted` 或 `rejected`；精度不达标属于可解释的拒绝结果，而不是程序崩溃。

`artifacts.py` 为每次运行创建唯一目录并原子写入产物。必须保存原始请求记录、校验后配置、采样元数据、分区成员、各模型验证指标、最终测试指标、可信模型 Pipeline、软件版本和哈希清单、模型卡及诊断图。

### 8.6 推理

`inference.py` 只加载本项目生成并登记的模型产物，加载前验证清单。输入 `gamma` 必须有限且位于训练范围内。返回值包含 `u(1)`、模型标识和运行标识。域外输入必须明确拒绝。

### 8.7 CLI

目标命令接口如下：

```text
surrogate-loop validate --config <path>
surrogate-loop run --config <path> --smoke
surrogate-loop run --config <path>
surrogate-loop report --run-dir <path>
surrogate-loop predict --run-dir <path> --gamma <value>
```

`--smoke` 只接受冒烟配置并核对其样本上限，不允许静默篡改正式配置。

## 9. 数据与控制流程

1. Codex 记录用户请求，并按固定 schema 生成或修改 JSON 配置。
2. `validate` 拒绝未知字段并完成确定性校验。
3. `run` 创建待执行运行目录，保存配置和环境信息。
4. 种子采样器生成互不重复的 `gamma` 工况。
5. 数值生成器计算 `u(1)`，并逐个使用解析解校验。
6. 工况被划分为训练、验证和测试分区。
7. 所有候选模型在训练集拟合，在验证集比较。
8. 冻结最佳验证模型，并在测试集上评估一次。
9. 验收门写入 `accepted` 或 `rejected` 状态并完成所有产物。
10. `predict` 验证产物与输入范围，重载模型并返回预测。

## 10. 失败处理

- 配置无效：在创建训练产物前非零退出，输出字段级错误。
- 求解器失败或解析校验失败：将运行标记为 `failed`，保存诊断，不训练模型。
- 单个候选模型失败：记录该候选错误；只要仍有有效候选即可继续。
- 所有候选模型失败：将运行标记为 `failed`，不生成最佳模型产物。
- 精度不达标：将运行标记为 `rejected`，保留最佳模型用于分析，并明确失败阈值。
- 产物损坏或清单不匹配：拒绝加载和预测。
- 域外或非有限预测输入：在调用模型前拒绝。
- 运行标识冲突：绝不覆盖旧运行，生成新的唯一标识。

## 11. 测试策略

实现阶段遵循测试驱动开发。测试分为三层：

### 11.1 单元测试

- 规范配置能够通过；未知字段、反向边界、非法样本数和未知模型被拒绝。
- 解析解满足 `gamma=0` 极限，并在负值、零、接近零和正值处与高精度数值积分一致。
- 固定种子生成相同样本。
- 三个数据分区互不相交。
- 域检查拒绝越界值、NaN 和无穷值。

### 11.2 集成测试

- 归一化器不在验证集或测试集上拟合。
- 模型注册表每个条目都支持训练、预测、保存和重载。
- 验证集负责选模，测试指标只为最终模型生成。
- 保存前后预测在浮点容差内一致。
- 清单损坏时模型加载被拒绝。

### 11.3 端到端测试

- 使用子进程执行 CLI 冒烟运行。
- 检查所有必需产物都已生成。
- 在新进程中重载模型并完成一次域内预测。
- 在新进程中验证域外预测会失败。

## 12. 环境方案

目标平台为 Windows 和 Python 3.11。使用 `uv` 管理依赖和虚拟环境，在仓库中建立 `.venv` 并提交 `uv.lock`。

运行依赖：

- NumPy
- SciPy
- scikit-learn
- Pydantic 2
- Matplotlib
- joblib

开发依赖：

- pytest
- pytest-cov
- Ruff

第一版只使用 CPU。本机 RTX 4060 不参与该闭环，因为当前候选模型不需要 GPU。只有后续经过批准的里程碑确实需要 PyTorch 时，才增加 CUDA 环境。

## 13. 完成标准

同时满足以下条件，才算完成第一版标量闭环：

1. 新克隆仓库可按中文 README 创建锁定环境并运行验证命令。
2. Ruff 和完整 pytest 测试通过。
3. CLI 冒烟运行成功并生成全部必需产物。
4. 数值标签在规定容差内通过解析解校验。

## 14. 实施顺序

实施计划必须把仓库可理解性放在首位，并按以下顺序拆解：

1. **仓库骨架和 README**：先创建文档、核心代码、算例、测试和运行产物目录，建立 `pyproject.toml`、`.gitignore`、`AGENTS.md`，并把根 README 更新为中文项目入口。
2. **锁定环境**：使用 `uv` 创建 Python 3.11 环境，加入最小运行/开发依赖并生成 `uv.lock`。
3. **配置合同和规范算例**：实现 Pydantic schema，并补齐正式与冒烟配置。
4. **解析解和数值数据生成**：以测试驱动方式实现科学计算内核。
5. **工况级划分和模型注册表**：实现防泄漏划分与三个模型后端。
6. **评价、选模和验收门**：固化指标、选模规则和测试集使用边界。
7. **产物持久化和有界推理**：完成可信重载、清单验证和域外拒绝。
8. **CLI 和端到端验证**：接通所有目标命令并运行完整冒烟测试。
9. **Codex 操作规则和复现文档**：完善中文指南，使自然语言入口能够稳定操作确定性 CLI。

完整时间轨迹 `gamma -> u(t)` 是本规格完成后的下一里程碑，不进入本轮实施计划。
