# Agent 协同与二维线弹性演示文档体系设计

## 1. 背景

仓库已经实现标量 ODE、一维热传导神经算子和二维线弹性神经算子三个闭环，并积累了中文设计规格、实施计划、操作指南、正式配置和本机实测结果。现有内容能够支撑开发与复现，但新 Agent 进入一个全新对话时仍面临以下问题：

- `AGENTS.md` 规定了开发纪律，但没有最小阅读顺序和任务路由。
- `docs/` 根目录同时包含稳定设计与数百至上千行的历史实施计划，新 Agent 容易过度读取已经完成的过程。
- README 面向普通用户，但不能完整承担 Agent 接管、演示和新 PDE 开发导航。
- 二维线弹性已有可展示的报告、图像、成本和速度证据，但缺少一套面向对话的演示剧本与进度协议。
- Smoke、Full、假求解测试和真实 FEniCSx 证据的边界虽然分别写在操作指南中，却没有汇总为新 Agent 的稳定入口。

本设计先整理仓库文档体系，建立可由新对话直接使用的 Agent 协作与演示知识入口。独立 Skill 只确定内容边界，暂不决定安装或分发形式。

## 2. 目标

整理后，新 Agent 应在不通读全部历史计划的情况下，快速完成以下任务：

1. 理解仓库目标、三个闭环、当前状态和功能边界。
2. 带领不了解神经网络的用户看懂二维线弹性代理模型演示。
3. 向技术受众准确展示控制方程、数据来源、模型结构、误差指标和证据等级。
4. 在从头运行闭环时持续报告明确进度、成本和当前证据。
5. 正确区分快速展示、现场运行、诊断和开发新 PDE 四类任务。
6. 在未来扩展新 PDE 时进入“中文规格评审—实施计划—测试驱动实现—分级验收”的开发流程。

文档完成标准是：新 Agent 在约三分钟内能够回答仓库解决什么问题、为什么选择二维线弹性演示、如何展示或从头运行、当前结果是否通过 Full，以及如何继续开发新 PDE。

## 3. 非目标

本阶段不实现：

- 新的 Web UI 或演示 CLI；
- 运行时自然语言解析或外部 LLM API；
- 任意 PDE 自动生成；
- 二维线弹性 Full 数据生成与确认性验收；
- 独立 Skill 的最终安装、发布或自动发现机制；
- 把已有长实施计划拆分、移动或归档；
- 复制运行产物到 Git。

## 4. 受众与演示定位

文档同时服务两类受众：

- 非专业受众需要理解有限元为什么昂贵、神经网络学习了什么、推理为什么快以及当前结果能说明什么。
- 技术受众需要看到控制方程、本构、边界条件、参数域、FEniCSx 数据来源、数据划分、物理门禁、基线、神经算子、误差和可复现命令。

演示以二维线弹性悬臂梁为主线。一维热传导可作为已经通过 Full 验收和可信推理的补充证据，标量 ODE 作为时间极短的备用闭环，但不分散二维线弹性主叙事。

## 5. 信息架构

### 5.1 目标结构

```text
AGENTS.md
README.md
docs/
├── README.md
├── 当前能力与状态.md
├── demos/
│   ├── README.md
│   └── 二维线弹性演示手册.md
├── guides/
│   ├── Agent协作指南.md
│   ├── 环境与验证.md
│   ├── 标量闭环操作指南.md
│   ├── 一维热传导闭环操作指南.md
│   └── 二维线弹性闭环操作指南.md
├── architecture/
│   └── README.md
└── 现有设计规格与实施计划
```

此前已确认规格和计划直接放在 `docs/` 根目录，不增加 `docs/superpowers/`。现有设计与实施计划不移动，避免破坏历史链接；`docs/README.md` 负责把它们归类为架构开发或历史追溯材料。

### 5.2 文档职责

`AGENTS.md`：

- 保存仓库强制规则；
- 增加新 Agent 最小阅读顺序；
- 增加四类任务路由入口；
- 强调 Full 授权、证据分级和运行产物边界。

根 `README.md`：

- 继续作为普通用户入口；
- 增加 Agent 接管和演示入口链接；
- 不复制完整演示剧本或当前实测表格。

`docs/README.md`：

- 作为文档总地图；
- 按演示、运行、诊断和开发新 PDE 路由阅读材料；
- 区分稳定文档与历史实施记录；
- 明确哪些文档不应默认全文读取。

`docs/当前能力与状态.md`：

- 汇总三个闭环的能力矩阵；
- 记录最近一次验证状态、证据等级和功能边界；
- 链接权威操作指南和配置；
- 不复制完整训练历史或大段设计内容。

`docs/guides/Agent协作指南.md`：

- 说明新对话如何接管仓库；
- 规定只读检查、任务分类、成本确认、进度播报和完成报告；
- 规定运行已有闭环、诊断和开发新 PDE 的不同路径；
- 规定 Agent 不能通过降低科学门槛或绕过摘要制造成功。

`docs/demos/README.md`：

- 提供演示入口；
- 说明二维线弹性主线、一维热传导补充和标量备用的选择逻辑；
- 区分快速展示与从头运行。

`docs/demos/二维线弹性演示手册.md`：

- 提供可直接用于新对话的演示剧本；
- 给出通俗解释、技术公式、六阶段流程和进度模板；
- 列出已有产物、从头运行命令、预计成本和失败降级；
- 明确 Smoke 只构成开发证据。

现有操作指南继续作为 CLI、环境、成本、运行产物和技术事实的权威来源。

### 5.3 默认阅读顺序

二维线弹性演示任务的最小阅读顺序固定为：

```text
AGENTS.md
→ docs/README.md
→ docs/当前能力与状态.md
→ docs/demos/二维线弹性演示手册.md
→ docs/guides/二维线弹性闭环操作指南.md
```

只有修改代码或扩展新 PDE 时，才继续阅读对应设计规格、实施计划和核心代码。新 Agent 不应在尚未分类用户任务前全文读取所有历史计划。

## 6. 二维线弹性演示设计

### 6.1 核心叙事

Agent 先用通俗语言解释：悬臂梁左端固定，右端载荷的材料、大小、方向和位置可以变化。FEniCSx 每次能够计算可信位移场，但重复求解需要秒级成本；代理模型通过预先生成的数据学习参数到完整位移场的映射，把后续预测缩短到毫秒级。

演示必须传达三个价值：

1. FEniCSx 生成经过物理门禁的训练标签。
2. Vector DeepONet 学习参数到连续坐标位移场的算子映射。
3. 模型训练后能显著加速重复查询，同时仍保留误差和证据边界。

### 6.2 技术公式

平衡方程为：

\[
-\nabla\cdot\boldsymbol{\sigma}(\mathbf u)=0.
\]

小应变为：

\[
\boldsymbol{\varepsilon}(\mathbf u)
=\frac12\left(\nabla\mathbf u+\nabla\mathbf u^{\mathsf T}\right).
\]

平面应力本构写为：

\[
\boldsymbol{\sigma}
=2\mu\boldsymbol{\varepsilon}
+\lambda_{\mathrm{ps}}
\operatorname{tr}(\boldsymbol{\varepsilon})\mathbf I.
\]

代理算子为：

\[
\mathcal G:(E,\nu,P,\theta,y_0,w)
\longmapsto \mathbf u(x,y)=(u_x,u_y).
\]

模型利用已知物理关系：

\[
\widehat{\mathbf u}(x,y)
=\frac{P}{E}\frac{x}{L}\,
\mathbf u_{\mathrm{raw}}(x,y).
\]

其中 `x/L` 保证左端固支，`P/E` 编码线弹性尺度关系。应力和 von Mises 图来自 FEniCSx 诊断，不是当前神经网络的正式输出。

### 6.3 快速展示模式

快速模式不重新训练，流程为：

```text
介绍问题与公式
→ 读取已有校准和 Smoke 报告
→ 展示位移真值、预测和误差
→ 展示 FEniCSx 应力诊断
→ 解释误差、加速比和物理门禁
→ 说明 Smoke 与 Full 的证据区别
```

预计用时为五至十分钟。Agent 必须验证产物存在且报告可读；若产物缺失或摘要损坏，不能继续引用相应结论。

快速展示默认面向当前保留了 `runs/` 实测产物的工作区。全新克隆不会包含被 Git 忽略的运行目录；此时 Agent 只能引用已提交操作指南中的实测摘要，并明确说明本地报告和图像尚不存在，或者在用户确认成本后执行一次新的 Smoke。不得为了便携演示而把完整运行目录、训练数据或模型提交到 Git。

### 6.4 从头运行模式

从头运行流程为：

```text
环境诊断
→ 配置校验
→ 可选执行校准
→ 生成 144 个 FEniCSx 样本
→ 训练 POD-RBF 和 Vector DeepONet
→ 生成开发评价、图像和速度报告
```

环境已配置时，完整二维 Smoke 在当前机器实测约五至六分钟；重新执行 16 样本校准还需约九分钟。Agent 必须在启动前说明时间、CPU/GPU、磁盘成本和 Smoke 证据等级。

若流水线复用了摘要一致的已有阶段，Agent 必须明确说明复用内容，不能把它描述为从零重新计算。需要证明完全从头执行时，应使用新的运行目录或新的受控运行身份，并在执行前说明原因。

### 6.5 六阶段进度协议

演示阶段固定为：

1. 了解问题与确认演示模式。
2. 环境和配置诊断。
3. FEniCSx 数据生成与物理门禁。
4. POD-RBF 与 Vector DeepONet 训练。
5. 开发测试评价、图像和速度基准。
6. 结论、限制和下一步。

从头运行期间使用统一格式：

```text
进度：3/6——生成有限元数据
已完成：环境诊断、配置校验
正在执行：144 个 FEniCSx 求解
预计耗时：约 5 分钟
当前证据：数据尚未发布，训练尚未开始
下一步：检查残差、力平衡和数据摘要
```

Agent 不得把“进程仍在运行”描述为“已经通过”。最终报告必须给出实际状态、证据路径、通俗指标解释和能否进入下一阶段的结论。

## 7. Agent 任务路由

### 7.1 新对话只读接管

Agent 首先执行：

```text
读取 AGENTS.md
→ 读取 docs/README.md
→ 读取 docs/当前能力与状态.md
→ 查看 git status 和最近提交
→ 将任务分类为演示、运行、诊断或开发
```

只读接管不授权安装依赖、生成正式数据、启动 Full、修改配置或提交代码。

### 7.2 演示

- 读取演示手册和二维线弹性操作指南。
- 询问快速展示或从头运行；默认推荐快速展示。
- 依次解释通俗问题、控制方程、代理映射、数据来源、模型和证据。
- 使用六阶段进度协议。
- 默认不运行 Full。

### 7.3 运行已有闭环

- 定位对应 `examples/` 配置和操作指南。
- 先校验环境与配置。
- 说明成本、设备、输出目录和验收语义。
- 从可信报告读取结果，不根据零散终端输出猜测状态。

### 7.4 诊断

- 先确定失败属于环境、求解器、协议、训练、验收还是推理。
- 保留失败状态和诊断产物。
- 不静默删除失败样本，不放宽 Full 门槛，不绕过 SHA-256，不把假求解测试作为科学证据。

### 7.5 开发新 PDE

开发路径固定为：

```text
自然语言需求
→ 明确控制方程、区域、边界、参数和输出
→ 识别可复用核心与新增领域代码
→ 编写中文设计规格并评审
→ 编写实施计划
→ 测试驱动实现
→ calibration / Smoke / Full 分级验证
```

新 PDE 不得通过把任意公式或代码直接写入配置并动态执行来实现。

## 8. 证据与授权边界

证据分为：

- 普通假求解 E2E：只证明编排、安全拒绝和接口兼容。
- 真实微型 FEniCSx E2E：证明跨环境协议和一次训练步骤可运行。
- calibration：证明求解器、网格、物理门禁和本机成本。
- Smoke：证明完整开发闭环，结果可以指导工程调整。
- Full：使用封存测试的一次性确认性验收。

Agent 必须准确使用这些名称。二维线弹性当前只完成 Smoke，不能通过正常可信推理入口伪装成 accepted Full 模型。启动 Full 必须重新获得用户明确授权；文档整理、快速展示和从头 Smoke 均不构成 Full 授权。

## 9. 失败降级

- FEniCSx 环境不可用：切换为已有产物快速展示，并明确现场求解未执行。
- CUDA 不可用：说明 CPU 回退及成本变化，不引用 GPU 训练耗时作为本次结果。
- Smoke 失败：展示失败阶段和诊断价值，不伪造完成报告。
- 时间不足：从头运行切换为快速展示，并明确发生了模式切换。
- 阶段被复用：列出复用的数据、模型或报告，不能称为完全从零运行。
- 图像或报告损坏：停止引用对应结论，不绕过完整性校验。
- 用户要求放宽 Full 门槛：说明这会改变科学合同，必须重新设计、评审并建立新的运行身份。

## 10. 文档质量门

实现后必须检查：

1. `AGENTS.md`、README 和文档地图中的本地链接有效。
2. 演示命令与当前 CLI 参数一致。
3. 配置路径、示例路径和已引用运行产物真实存在。
4. 公式与二维线弹性设计规格及实现一致。
5. 动态实测数值只在 `当前能力与状态.md` 和权威操作指南中维护，演示手册通过链接引用。
6. Smoke、Full、假求解和真实 FEniCSx 的证据语义没有混淆。
7. 快速展示不依赖重新训练。
8. 从头运行发生恢复或复用时，Agent 会明确说明。
9. 用户可读新增文档全部使用中文。
10. 文档整理不改变已有科学配置、运行产物或核心代码行为。

## 11. Agent 接管演练

使用三个新对话场景进行文档验收：

1. “介绍这个仓库并演示二维线弹性的价值。”
2. “从头运行二维线弹性闭环，并持续告诉我进度。”
3. “我要扩展一个新的 PDE 代理模型。”

合格的新 Agent 应在不全文读取所有历史计划的情况下，准确选择入口、说明边界并给出下一步。演练可以由人工按检查表执行；本阶段不要求自动启动额外 Agent。

## 12. 独立 Skill 内容边界

未来 `surrogate-loop-demo` Skill 应包含：

- 触发场景识别；
- 最小文档读取顺序；
- 演示、运行、诊断和开发四类任务路由；
- 二维线弹性六阶段演示；
- 公式讲解顺序；
- 进度播报模板；
- Smoke/Full 授权边界；
- 常见失败的诊断路由；
- 完成报告格式。

Skill 不包含：

- 大段设计规格；
- 固定实测数值；
- 完整 CLI 文档；
- Python 或 FEniCSx 实现细节；
- 绕过仓库校验的命令。

动态事实必须从 `docs/当前能力与状态.md`、操作指南、配置和运行报告读取。文档整理并完成新对话演练后，再评估 Skill 采用仓库内模板、个人安装或其他分发形式。

## 13. 实施边界

本设计实施时只允许：

- 新增文档地图、能力状态、Agent 协作指南和演示手册；
- 更新 `AGENTS.md`、根 README 和必要导航链接；
- 增加文档链接或命令一致性检查；
- 记录演练结果。

不修改模型、求解器、配置门槛、训练数据和运行产物。发现文档与实现不一致时，先把差异记录为问题；若修复需要改变代码行为，必须进入独立规格与实施流程。
