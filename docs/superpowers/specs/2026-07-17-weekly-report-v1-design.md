## 2. 目标

在 `docs/` 下建立可持续维护的周报目录，并以第一期周报完整说明：

1. 仓库解决什么问题，以及三个代理模型闭环如何逐级演化；

报告必须让管理层能够快速理解工作价值，同时允许技术人员沿证据链复核数字和状态。

## 5. 报告叙事结构

第一期周报采用“管理层主线 + 单文件技术证据附录”。正文关键数字可以独立阅读，证据附录提供逐项核查路径。

### 5.1 标题与 Executive Summary

标题为“代理模型训练闭环项目周报（第 01 期）”。紧随标题提供明确标注的 `Executive Summary`，用 3–4 条结论回答：

- 本周完成了什么；

### 5.2 仓库目标与基本架构

用管理层可理解的语言说明仓库不是任意方程执行器，而是自然语言协作层驱动白名单、确定性科学程序的本机闭环。

使用 Mermaid 表示主要数据流：

```mermaid
flowchart LR
    A["自然语言需求与人工确认"] --> B["白名单配置"]
    B --> C["数值求解与数据门禁"]
    C --> D["代理模型训练与验证选模"]
    D --> E["模型冻结与分级验收"]
    E --> F["摘要保护的域内推理"]
```

架构介绍覆盖：

- 自然语言协作层；
- 配置与问题模板层；
- 数值求解和数据协议层；
- 代理模型与基线层；
- Smoke/Full 证据分级；
- 运行身份、摘要和可信推理层。

### 5.3 三个闭环的能力递进

通过一张对比表呈现：

| 层级 | 问题 | 输入/输出类型 | 主要模型 | 当前证据 |
|---|---|---|---|---|
| 1 | 强迫反应标量 ODE | 参数到标量 | PRS/GPR/MLP 选模 | accepted |
| 2 | 一维热传导 | 参数与坐标到连续标量场 | DeepONet | Full accepted |
| 3 | 二维线弹性 | 参数与二维坐标到向量场 | Vector DeepONet | Full accepted |

该部分强调能力从“预测一个值”发展到“学习连续物理场”，再发展到“通过外部有限元求解器学习二维向量场”。

### 5.4 核心算法

核心算法按“解决了什么问题”组织，而不是按代码模块罗列：

1. 标量闭环：PRS、GPR、MLP 候选模型通过验证集选模；
2. 一维热传导：Crank–Nicolson 生成标签，解析解验证数值误差，POD/GPR 作为基线，DeepONet 学习连续时空场；
3. 二维线弹性：FEniCSx/PyAMG 生成平面应力位移标签，制造解、残差、力平衡、载荷线性和网格收敛构成物理门禁；
4. `directional_linear_v2`：Branch 学习 `(nu,y0,w)`，Trunk 学习 `(x,y)`，输出水平/竖向响应基，再精确执行方向叠加、`P/E` 缩放和固支边界约束；
5. 可信验收：训练集拟合归一化，验证集选模，Full 在冻结后一次性消费 sealed-test。

只保留帮助理解的公式和维度，不展开完整有限元弱形式或训练实现细节。

### 5.5 二维线弹性主线

该部分占报告正文约 70%，按问题—诊断—改进—确认的顺序展开：

1. 固定矩形悬臂梁、六个物理参数和二维位移输出；
2. calibration 证明网格、求解器和物理门禁可用；
3. 初始 Smoke 暴露近水平载荷方向误差；
4. `directional_linear_v2` 利用线性叠加结构解决方向泄漏；
5. 同一批 144 个 FEniCSx 样本上的改进对照；
6. Full 新生成 512/96/128 个训练、验证、封存测试样本；
7. 三个固定种子训练、验证选模、冻结和一次性封存验收；
8. 最终状态 `accepted`，正常推理入口成功加载。

### 5.7 当前能力边界

报告将仓库定位为“固定白名单物理模板下的本机研究级可信代理模型闭环”，并明确区分：

已经具备：

- 标量、连续一维标量场和连续二维向量场建模；
- 内置求解器与外部 FEniCSx 数据源；
- 确定性采样、训练/验证/测试隔离；
- calibration、Smoke、Full 分级证据；
- 模型冻结、一次性封存测试和受保护推理；
- 配置、数据、模型、报告和阶段产物摘要链。

尚不具备：

- 任意 PDE、任意几何或仅通过 JSON 注册新物理问题；
- 三维、非线性、接触、断裂、塑性或动力弹性；
- 训练域外泛化保证、校准不确定性或主动学习；
- 实验数据验证和模型形式误差校准；
- Web 服务、多用户、模型注册中心、分布式调度或生产 SLA；
- 防止能够同时重写文件和摘要的恶意主体；
- 神经网络应力、应变和 von Mises 的工程验收精度。

## 6. 证据链设计

### 6.1 事实来源优先级

报告按以下优先级采用事实：

1. 单次 accepted 运行中的状态、报告和摘要；
2. 当前可执行配置；
3. 稳定操作指南和状态页；
4. 设计规格；
5. 历史实施计划仅用于解释决策过程，不作为当前状态来源。

若文档与运行产物冲突，以通过严格读取入口验证的运行产物为准，并同步修正文档。

### 6.2 二维线弹性证据路径

第一期周报附录使用以下链路：

```text
request.json
→ spec.json / full.json
→ sample_plan.json / solver_job.json
→ dataset_manifest.json / solver_quality.json
→ training_candidates.json
→ freeze_manifest.json
→ sealed_test_summary.json
→ acceptance_stage.json
→ acceptance.json
→ elasticity2d predict
```

每个节点说明：

- 它证明什么；
- 它不能证明什么；
- 与前后节点通过哪个摘要或身份关联；
- 本地相对路径在哪里。

关键结论使用“结论—指标—门槛—来源”表映射，避免只堆积文件名。

### 6.3 其他两个闭环证据

- 标量 ODE：`manifest.json`、`test_metrics.json` 和正常 `predict`；
- 一维热传导：严格 `manifest.json`、Full test metrics、模型卡和正常 `operator predict`。

报告不会声称三套产物具有完全相同的封存实现；只说明它们各自已经达到仓库为该闭环定义的 accepted 证据等级。

### 6.4 本地运行目录边界

`runs/` 被 Git 忽略。周报中的关键数字必须在正文保留，链接用于本机核查而不是承担唯一信息来源。全新克隆若没有本地运行目录，读者仍能理解结论，但不能仅凭 Git 仓库重新验证本机 accepted 产物摘要。

## 7. 状态同步规则

稳定文档更新必须满足：

- 把二维线弹性最高证据从 Smoke `development_complete` 更新为 Full `accepted`；

- 测试计数只记录有新鲜命令输出支持的数字。

## 8. 错误与缺失证据处理

- 若本地证据文件缺失，周报标记“本地证据不可用”，不猜测摘要或路径；
- 若严格报告入口拒绝运行产物，不能把它写成 accepted；
- 若文档链接无法解析，修复导航后再交付；
- 若某个指标没有冻结门槛，不使用“通过”措辞，只描述观测值；
- 若 Mermaid 无法渲染，正文必须仍能通过相邻文字理解架构；
- 不复制 sealed-test 原始数组到文档或 Git。
