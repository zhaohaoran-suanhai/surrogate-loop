# 二维方腔 Navier–Stokes 代理模型闭环设计

## 1. 设计状态

- 日期：2026-07-23
- 状态：用户已批准
- 主要仓库：`surrogate-loop`
- 高保真求解仓库：`../fluent-automation`
- 高保真求解器：Ansys Fluent 2024 R1 / PyFluent 0.37.0
- 正式代理模型：POD-RBF
- 首版目标：在固定二维顶盖驱动方腔中，根据域内 Reynolds 数预测完整稳态流场

## 2. 建设目标

在现有代理模型训练闭环中增加第四个白名单问题：

```text
Re
→ [u(x,y;Re), v(x,y;Re), p'(x,y;Re)]
```

其中高保真标签由本机 `fluent-automation` 仓库通过受信 Runner、PyFluent 和 Fluent
2024 R1 生成；`surrogate-loop` 负责确定性采样、数据隔离、POD-RBF 训练、验收、报告和
受保护推理。

首版必须证明：

1. 自然语言需求可以收敛为固定问题模板和严格配置；
2. 多个 `Re` 工况可以由 Fluent 可靠、可审计地生成；
3. 训练、验证和测试按完整 Reynolds 数工况隔离；
4. POD-RBF 能对 `Re ∈ [10,400]` 内未见过的工况预测完整 `u、v、p'` 场；
5. 模型、数据、Fluent 证据和推理结果可以通过身份与 SHA-256 清单追溯；
6. 域外、非有限、状态不可信或产物损坏的推理请求会被明确拒绝。

## 3. 非目标

首版不包括：

- 任意二维或三维 Navier–Stokes 方程；
- 任意几何、障碍物、入口、出口或多腔体；
- 改变方腔长宽比；
- 改变顶盖速度分布或让其他壁面运动；
- 瞬态、周期、非定常或湍流方腔；
- `Re < 10` 或 `Re > 400` 的可信外推；
- U-Net、FNO、DeepONet、PINN 或坐标 MLP；
- 让 `surrogate-loop` 直接启动 `fluent.exe` 或 PyFluent；
- 把 Fluent 内部离散残差与仓库独立计算的物理诊断混为一谈；
- 将 Smoke 结果表述为 Full 确认性验收或生产级 CFD 认证。

## 4. 固定物理合同

问题标识固定为：

```text
fluent_lid_driven_cavity_steady_v1
```

物理域为单位正方形：

\[
\Omega=[0,1]\times[0,1].
\]

求解稳态、二维、不可压缩、层流 Navier–Stokes 方程：

\[
\nabla\cdot\mathbf u=0,
\]

\[
(\mathbf u\cdot\nabla)\mathbf u
=-\nabla p+\frac{1}{Re}\Delta\mathbf u.
\]

边界条件固定为：

```text
top_moving_wall:      u = 1, v = 0
bottom_wall:          u = 0, v = 0
stationary_side_wall: u = 0, v = 0
```

Fluent 中使用无量纲等价设置：

```text
density   = 1
lid speed = 1
viscosity = 1 / Re
Re range  = [10,400]
```

压力使用零均值规范：

\[
p'=p-\frac{1}{|\Omega|}\int_\Omega p\,d\Omega.
\]

正式模型和报告只比较 `p'`，不学习或评价无物理意义的压力常数偏移。

## 6. 跨仓库架构

两个仓库不互相导入 Python 包，只交换版本化 JSON/NPZ 文件：

```text
surrogate-loop
  cavity2d plan
  ├─ 校验固定物理合同
  ├─ 确定性生成 Re 样本
  ├─ 预先确定 train/validation/sealed-test
  └─ 写 solver_request.json
                │
                ▼
fluent-automation
  lid_driven_cavity.pipeline
  ├─ 校验请求、网格哈希和白名单
  ├─ 将样本分为固定小批次
  ├─ 一次只生成一个受信 job
  └─ 汇总批次证据
                │
                ▼
scripts/windows/run-ansys-job.ps1
  → Runner
  → PyFluent 0.37.0
  → Fluent 2024 R1
                │
                ▼
fluent-automation
  per-sample fields / diagnostics / case-data
  → batch acceptance
  → reload audit
  → pipeline-complete.json
                │
                ▼
surrogate-loop
  cavity2d import/train
  → FieldDataset
  → POD-RBF
  → development or Full acceptance
  → protected inference
```

职责边界如下。

### 6.1 `fluent-automation`

负责：

- 受信 Fluent 启动、退出和进程归属；
- 固定方腔网格、区角色和物理设置；
- SIMPLE 求解、初始化、迭代与收敛；
- 坐标、速度、压力、残差和运行诊断导出；
- case/data、transcript、launch report、acceptance 和独立重载；
- 批次级 append-only 结果与恢复控制器。

不负责：

- POD-RBF 训练；
- 训练、验证和 sealed-test 的模型使用；
- 代理模型推理或模型验收。

### 6.2 `surrogate-loop`

负责：

- 固定 NS 配置合同和确定性 Reynolds 数采样；
- train/validation/sealed-test 预先划分；
- Fluent 数据清单、哈希和分区身份验证；
- 零均值压力、POD、RBF、评价和报告；
- Full 一次性 sealed-test 状态机；
- accepted 模型的域内推理。

不负责：

- 直接调用 Fluent、PyFluent 或 Ansys 可执行文件；
- 修改 Fluent 许可证、系统服务或用户进程；
- 在 Fluent 证据未通过时构造伪数据继续训练。

## 8. Fluent 求解合同

每个样本固定执行：

1. 读取并校验已批准网格；
2. 检查 `fluid` 和三个壁面区的名称与类型；
3. 设置二维、双精度、压力基、稳态、层流；
4. 设置 `density=1`、`viscosity=1/Re`；
5. 设置顶部切向速度为 1，其余壁面静止无滑移；
6. 设置 SIMPLE、standard pressure 和 second-order-upwind momentum；
7. 对该工况独立执行 Hybrid Initialization；
8. 最多迭代 5000 次；
9. 导出坐标、`u、v、p`、残差和诊断；
10. 写出样本 case/data 和结构化 acceptance。

每个工况都从独立初始化开始，不使用前一个 `Re` 的场作为未声明热启动。

样本成功必须同时满足：

- 问题、网格、区角色和 Fluent 设置与合同完全一致；
- continuity、x-momentum、y-momentum 最终缩放残差均不高于 `1e-6`；
- 坐标、速度、压力和诊断全部有限；
- 坐标数量与网格单元数量一致；
- 没有发散或 fatal 标记；
- 迭代数不超过 5000；
- 字段、case/data、transcript 和 acceptance 文件存在且非空。

任一门槛失败时，该样本失败；失败样本不得静默删除、替换或进入训练。

## 9. 分块执行与恢复

除垂直切片外，固定批次上限为 8 个 Reynolds 数：

```text
一个受信 job
→ 启动一个 Fluent 会话
→ 依次独立初始化并求解最多 8 个 Re
→ 保存样本级检查点与产物
→ 汇总 batch acceptance
→ 正常退出并执行进程审计
```

Runner 保持单任务串行，不并发争用许可证。

批次采用原子验收：

- 任一样本失败，整个批次状态为 `failed`；
- 已写出的样本诊断仍保留；
- 失败批次不能进入最终 dataset manifest；
- 重试使用新的输出目录并重算整个批次；
- 不跨失败批次复用部分样本；
- 只有全部样本通过的批次才能被 pipeline 汇总。

控制器沿用已验证的 Batterie 模式：只读取证据并生成下一份 job，不导入 PyFluent、不启动
Ansys。所有批次、独立重载、退出和进程审计通过后，才写 `pipeline-complete.json`。

## 10. 样本计划

采样在 `log10(Re)` 空间进行确定性分层，所有集合按完整 Reynolds 数工况隔离。

### 10.1 垂直切片

```text
Re = 100
```

只证明 2D Fluent 启动、方腔设置、收敛、字段导出、case/data、独立重载和跨仓库协议。

### 10.2 Calibration

```text
Re = 10, 100, 400
```

用于验证低、中、高端工况的求解稳定性、迭代成本、字段尺度、磁盘规模和物理诊断量级。
Calibration 不训练模型。

### 10.3 Smoke

```text
train       = 16
validation  = 4
development_test = 4
seed        = 2026072301
```

Smoke 用于工程闭环、POD/RBF 选择和预算校准，最高状态为 `development_complete`。

### 10.4 Full

```text
train       = 80
validation  = 20
sealed_test = 20
seed        = 2026072302
```

Full 的 120 个工况全部独立于 Smoke。sealed-test 的场数据在模型、缩放器、POD、RBF 参数、
验收阈值和推理代码冻结前不能进入训练进程。Full 必须在 Calibration 和 Smoke 评审后获得
新的明确授权。

## 11. 数据协议

协议固定包含：

```text
schema_version
problem_id
solver_request_id
mesh_sha256
fluent_version
pyfluent_version
sample_plan_sha256
batch_id
sample_ids
split_roles
field_archive_sha256
diagnostics_sha256
case_data_sha256
```

每个通过的批次输出安全 NPZ：

```text
sample_ids   [n_samples]
parameters   [n_samples, 1]          # Re
coordinates  [n_cells, 2]            # x, y
fields       [n_samples, n_cells, 3] # u, v, p'
```

要求：

- 不允许 pickle 或对象数组；
- 坐标按 `(y,x)` 确定性排序；
- 每个样本使用同一排序映射；
- 所有批次的坐标数组和 SHA-256 必须完全一致；
- `p'` 逐样本减去单元平均压力；
- 分区角色在 Fluent 求解前固定，不能由结果决定；
- 所有 JSON 拒绝未知字段；
- 协议版本、问题 ID、网格摘要、形状、数据类型或 SHA-256 不匹配时，训练侧拒绝导入。

每个批次至少选择一个代表性样本，在全新 Fluent 会话中重载 case/data，验证设置、迭代位置、
场数据可读、退出确认和零残留进程。所有样本都保留字段、收敛诊断和 case/data；代表性重载
不替代其他样本的求解门禁。

## 12. POD-RBF 设计

输入特征为：

\[
z=\operatorname{normalize}(\log_{10}Re).
\]

模型分为两条支路：

```text
[u,v] → velocity POD → velocity RBF coefficients
p'    → pressure POD → pressure RBF coefficients
```

速度和压力分开降维，避免压力尺度改变速度模态选择。

训练集独占：

- `log10(Re)` 均值和标准差；
- 速度场均值和 POD 基；
- 压力场均值和 POD 基；
- RBF 拟合。

候选合同固定为：

```text
POD energy threshold: 0.999 or 0.9999
maximum components:   64 and <= n_train - 1
RBF kernel:           cubic, thin_plate_spline, multiquadric
RBF smoothing:        0, 1e-10, 1e-8
```

验证集先按速度相对 L2 排名，再按压力相对 L2 和模型复杂度进行确定性平局裁决。测试集不参与
POD、RBF、kernel、smoothing 或模态数选择。

模型产物只保存 JSON 和安全数值数组，包括问题合同、输入缩放、场均值、POD 基、RBF 训练点、
RBF 系数与核参数。不得加载任意外部 pickle、joblib 或完整 Python 模型对象。

## 13. 评价与 Full 验收

Full 的 20 个 sealed-test 工况必须同时满足：

| 指标 | 门槛 |
|---|---:|
| 速度相对 L2 中位数 | ≤ 2% |
| 速度相对 L2 P95 | ≤ 5% |
| 速度最差样本相对 L2 | ≤ 10% |
| 零均值压力相对 L2 中位数 | ≤ 5% |
| 零均值压力相对 L2 P95 | ≤ 10% |
| 零均值压力最差样本相对 L2 | ≤ 20% |
| 主涡中心位置误差 P95 | ≤ 0.05 个方腔边长 |
| 所有预测 | 有限 |
| CPU 推理相对单样本 Fluent | ≥ 100× |

报告还必须包含：

- 水平和竖直中心线速度误差；
- 速度模、压力和误差云图；
- 流线和主涡中心；
- 仓库独立计算的离散散度；
- 仓库独立计算的离散动量残差；
- 单样本 Fluent 和 POD-RBF CPU 时间；
- 最差速度、压力和物理诊断工况。

散度和动量残差明确标记为统一观测网格上的代理场物理诊断，不表述为 Fluent 原生离散残差。
首版记录其分布；Calibration 完成前不把尚未建立数值尺度的诊断量设置为 Full 硬门槛。

状态语义：

```text
calibration_complete  求解器与协议校准通过
development_complete Smoke 工程闭环完成
accepted             Full 所有冻结门槛通过
rejected             Full 完成但代理模型未达门槛
failed               配置、求解、协议、训练或产物失败
```

`rejected` 和 `failed` 保留完整诊断，但正常推理入口拒绝加载。

## 14. 推理合同

目标接口：

```powershell
uv run surrogate-loop cavity2d predict `
  --run-dir runs/<accepted-full> `
  --re 150 `
  --output predicted-flow.npz
```

推理前必须验证：

- 运行状态为 `accepted`；
- `problem_id`、模型 schema 和网格身份匹配；
- 配置、数据、模型和报告 SHA-256 清单完整；
- `Re` 有限且位于闭区间 `[10,400]`；
- 坐标、POD、RBF 和输出形状一致。

输出 NPZ 包含：

```text
reynolds
coordinates [n_cells,2]
velocity    [n_cells,2]
pressure    [n_cells]   # p'
```

报告或可选后处理生成：

- `u、v` 和速度模云图；
- 零均值压力云图；
- 流线；
- 水平和竖直中心线曲线；
- 主涡中心位置；
- 推理耗时和模型身份。

域外 Reynolds 数、不同几何、不同顶盖、瞬态、湍流或其他方腔请求必须明确拒绝，不能自动
外推或切换为其他物理模板。

## 15. 失败处理

### 15.1 Fluent 侧

- 请求、schema、网格或区角色无效：启动 Fluent 前拒绝；
- 2D 启动、许可证、连接或权限失败：按 Runner 阶段分类并保留 launch report；
- 求解未达残差门槛：样本和批次失败，不导出为训练样本；
- NaN/Inf、字段缺失或坐标不一致：批次失败；
- case/data 或 transcript 缺失：批次失败；
- 退出或进程审计失败：批次失败；
- 重试创建新的结果目录，不覆盖旧证据。

### 15.2 模型侧

- manifest、摘要、形状或数据身份不匹配：导入前拒绝；
- 训练失败：状态为 `failed`，不打开 sealed-test；
- Full 打开 sealed-test 后不得在同一运行身份继续调参；
- Full `rejected` 后修改模型必须创建新身份和新的 sealed-test；
- 模型清单损坏、状态不可信或输入域外：加载前拒绝。

## 16. 测试策略

两个仓库都遵循测试驱动开发：先写失败测试，再写满足规格的最小实现。

### 16.1 `fluent-automation` 单元测试

- registration 只注册固定 workflow、mode、输入和参数；
- 额外字段、错误批次、域外 Re 和错误网格摘要被拒绝；
- 固定 2D double-precision 启动参数不能由 job 改写；
- `viscosity=1/Re`；
- 顶盖、静止壁面、层流、SIMPLE 和离散格式设置准确；
- 每个样本均独立初始化；
- 字段形状、坐标排序和零均值压力准确；
- 残差超限、NaN、区缺失和文件缺失导致失败；
- pipeline 一次只生成一个 job，且不导入或启动 PyFluent；
- 假会话验证 API 调用顺序和关闭路径。

### 16.2 `fluent-automation` 真实验证

5. 全新 Fluent 会话独立重载；
6. `exit_confirmed=true`；
7. `remaining_pids=[]`。

普通 pytest 不启动 Fluent。真实验证必须使用唯一 Runner，并在执行前说明成本、输出目录和
授权边界。

### 16.3 `surrogate-loop` 单元测试

- 严格配置、固定问题 ID 和域边界；
- log-Re 采样可复现且分区无重叠；
- manifest、SHA-256、坐标和字段协议验证；
- 压力逐样本零均值；
- POD 和缩放只拟合训练集；
- RBF 候选选择和平局规则确定；
- 指标、主涡定位、域外和非有限输入拒绝；
- 安全模型数组保存和重载。

### 16.4 `surrogate-loop` 集成与 E2E

- 使用合成 Fluent 协议数据完成导入、训练、评价和报告；
- Smoke 不获得 `accepted` 状态；
- sealed-test 在冻结前不可读取；
- Full 指标决定 `accepted` 或 `rejected`；
- 模型数组、manifest 或坐标摘要损坏时加载失败；
- 子进程 CLI 完成训练，并在新进程完成一次域内预测；
- 域外预测返回稳定非零退出码；
- 现有标量、热传导和线弹性闭环保持兼容。

3. 在两个仓库建立隔离工作区并保护现有工作树；
4. 在 `fluent-automation` 以测试驱动方式实现方腔案例、注册和控制器；

## 18. 完成标准

同时满足以下条件，首版才算完成：

1. 两个仓库的模块边界、测试和文档与本规格一致；
2. `Re=100` 受信 Fluent 垂直切片和独立重载通过；
3. Calibration 的三个工况通过求解门禁；
4. Smoke 数据生成、POD-RBF 训练、报告和域内预测完整；
5. Full 使用独立 80/20/20 数据和一次性 sealed-test；
6. Full 达到全部冻结速度、压力、涡中心和加速门槛；
7. accepted 模型能在新 Python 进程预测任意域内 `Re`；

9. Fluent 请求、批次、字段、case/data、重载和进程证据可审计；
10. 现有三个代理模型闭环和三个 Fluent 自动化案例保持兼容；
